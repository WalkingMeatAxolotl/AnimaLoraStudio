Hotfix: ARB bucketing no longer drops images from small buckets + the per-epoch step count is computed per bucket

### Fixed

- **Fixed a bug where an ARB bucket's tail was dropped whenever it held fewer than batch_size images (short batches are now kept, matching kohya / ai-toolkit)**
  `BucketBatchSampler` defaulted to `drop_last=True`, so `len(bucket) % batch_size` images were dropped from every ARB bucket. Worst of all, **a bucket smaller than batch_size was dropped entirely** — at batch_size=2, for example, every bucket holding a single odd-aspect-ratio image would never be trained on.

  Tracing the git history: `drop_last=True` was written into the very first ARB version (commit `6774079`, "feat: add arb", 2026-02-08) with no commit message or ADR explaining the intent — it was likely carried over from PyTorch `DataLoader`'s sanitary default. But in the ARB case the trade-off has flipped: a plain DataLoader drops one remainder per epoch, whereas ARB drops one remainder per bucket, so the loss scales linearly with the number of buckets.

  - `phases/dataset.py`: `drop_last=False`, matching the "keep short batches" semantics of kohya `sd-scripts` (`math.ceil(len/bs)`) and ostris `ai-toolkit` (`min(end_idx, len(bucket))`)
  - Diffusion DiTs use LayerNorm/GroupNorm and are insensitive to dynamic batch sizes; `loop.py` already reads the batch size dynamically from `latents.shape[0]`, so a short batch introduces no numerical issues
- **The per-epoch step count is now computed exactly per ARB bucket (it was previously too low, leaving the LR scheduler misaligned near the end)**
  `__len__` previously used the global `n // batch_size` (or `ceil(n/bs)`), but the actual ARB batch count is `Σ_b f(n_b, bs)`, since each bucket has its own remainder. The global formula underestimated the batch count → `steps_per_epoch` was off → the scheduler's `total_steps` was off → the learning rate was misaligned over the final few steps.

  For example: a dataset of 129 images with batch_size=2 logged "steps per epoch: 64" (= 129 // 2), but the actual per-bucket floor sum comes out smaller (each bucket loses one remainder). The fix iterates over `bucket_for_index` and sums the per-bucket floor/ceil counts.

  Added `tests/test_bucket_batch_sampler.py` covering: small buckets aren't dropped, the historical drop_last=True behavior, `__len__` computed per bucket, `__len__` equaling the number of batches actually yielded, and falling back to the global formula when no bucket info is available.
