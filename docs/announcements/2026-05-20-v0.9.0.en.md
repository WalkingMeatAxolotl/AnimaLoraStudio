Training adds pause / resume + huber loss + a mixed_uniform timestep mode + automatic trigger-word injection into captions + a fully bilingual (English / Chinese) front end

### Added

- **Pause / resume training from the UI + queue hold scheduling (ADR 0006) (#100, #101, #105)**
  Four new interactions on the Queue / QueueDetail pages (see the full [ADR 0006](docs/adr/0006-queue-pause-resume.md) plus Addendum 1 and the design doc, after three rounds of review):

  - **Pause a task**: pausing a running task first opens the PauseConfirmModal to explain the semantics (per Addendum 1, pause = cancel + immediate GPU release, and the resume point is the most recent end-of-epoch auto-backup). After you confirm, the child process receives `CTRL_BREAK_EVENT` / `SIGINT`; handle_interrupt saves the LoRA as "interrupted" and writes a pause snapshot that freezes the current args → the child exits 0 → the task is marked paused. Throughout, the UI guides you with the PauseProgressModal (saving / a three-way choice on a 30s timeout / failure fallback).
  - **Resume a task**: resuming a paused row sets its status back to pending → on the next dispatch the supervisor launches it with `--resume-state <auto_epoch_state.pt>`, and bootstrap_phase reads the sibling config snapshot to override the args (the snapshot is the frozen source of truth from ADR §5.7, fully decoupled from any later edits you make to the version / preset / yaml) → a successful load_training_state emits `resume_state_loaded` → the supervisor clears the pause file pair.
  - **Hold / resume queue scheduling**: a sticky banner at the top; when a running task is present, the hold confirmation modal adds a "let it finish" / "pause it too" choice, and the primary button label updates with the radio selection.
  - **Cancel a paused task**: paused rows offer "cancel permanently," which deletes the pause files and clears the db fields.

  Signal path (verified end to end by the #96 spike): supervisor `CTRL_BREAK_EVENT` / POSIX `SIGINT` → child SIGBREAK / SIGINT handler → handle_interrupt saves state and emits `__EVENT__:pause_state` → the supervisor's `_finish_slot` branches across paused / canceled / done / failed, falling back to canceled when pause_pending is set but state_path is empty. db migration v6 adds four NULLABLE columns plus a `queue_settings` kv table to persist the hold switch (no backfill needed for old databases).
- **The Tagging page gains a trigger-word input that is written into every caption and sample-image prompt once tagging starts (#103)**
  Ends the "forgot the trigger word and wasted a run" footgun. A compact input is added at the top of Step 4 (Tagging); when tagging starts, the trigger is saved to `versions.trigger_word` (migration v7), and the tag worker injects the trigger as the first tag in every caption:

  - `.txt` / simple list `.json`: trigger becomes `tags[0]` (deduplicated case-insensitively)
  - LLM `documented_full` `.json`: injected as a top-level `meta.trigger` (without polluting the `ai_output` / `fixed` / `character` business fields)
  - caption_text mode: prepends `f"{trigger}, {text}"`
  - The training runtime's `bootstrap.py` injects the trigger into sample_prompt automatically, so sample images naturally include the trigger word (token-level case-insensitive matching, so "art" won't be mistaken for "artist")
  - The TagEdit page shows a read-only trigger badge in its actions area to shorten the cognitive distance of "what was the trigger again?"

  Compared with other mainstream trainers: Civitai's online trainer makes it mandatory at project creation → forced; AI-Toolkit does a `[trigger]` runtime substitution → not portable; this PR takes the "normalize at authoring time" route, where the caption files are the source of truth and travel with you if you switch trainers.
- **The entire Studio front end is now bilingual (English / Chinese), with a language picker on first launch (#76, #89)**
  - **#76**: the Regularization / Settings / Generate / Testing / Presets / Project Overview / LLM Tagger pages are fully localized; schema-exposed fields now use frontend-owned i18n keys (backend enums / field names stay stable while the UI renders localized labels, descriptions, and disable hints); the Settings model / upscaler directory descriptions and the LLM Tagger builtin presets / message-editor copy are localized too; schema-driven takeover is restored and extended (PPSF disables the scheduler / InfoNoise disables timestep / Prodigy and PPSF lock LR=1).
  - **#89**: the first-launch language choice moves from a `studio.bat/sh` CLI prompt (cmd.exe's default codepage can't display Chinese characters) to a front-end modal. It pops up automatically when `localStorage['studio.lang']` is null, showing two cards — "English / 中文" — each with a subtitle in its own language; keyboard focus initially lands on the card matching `navigator.language` without steering you visually. Fresh installs no longer have a blocking prompt, so CI and unattended deployments pass straight through.
- **Training adds a huber loss option that is more robust to extreme samples (#75, #86, #87)**
  - The schema adds `loss_type: Literal["mse", "huber"]` plus a dedicated `huber_c` field; the default is `mse`, and old yaml / presets behave bit-for-bit identically (a test codifies that MseLoss.compute is strictly equal to `F.mse_loss(reduction='none')`).
  - Implemented as a sub-package following the ADR 0003 plugin registry pattern: the BUILDERS + build_loss + validate_schema_consistency trio plus a `LossProtocol` (a runtime_checkable Protocol).
  - **HuberLoss** is simplified to a single constant δ (#86 removes the snr/sigma schedule introduced in PR #75 — mathematically it degenerated into MSE and had no paper or mainstream-trainer basis).
  - InfoNoise's `_raw_mse` is decoupled from the training loss and always computes a separate `F.mse_loss` (so that enabling huber doesn't break the I-MMSE assumption from the InfoNoise paper), with `test_infonoise_raw_mse_decoupled_from_loss_type` guarding this boundary.
- **New mixed_uniform timestep sampling mode + a post-sampling schedule shift (#73, #85)**
  - The `timestep_sampling` Literal gains two modes, `mixed_uniform_low` / `mixed_uniform_logit`, which start from full uniform coverage and mix in the biased-end distribution of `logit_normal_low` / `logit_normal` at the `timestep_mix_low_prob` ratio.
  - New field `timestep_schedule_shift: float = 1.0`: unlike `timestep_shift` (internal to logit-normal), this applies one extra shift to t **after** sampling, `t' = (t·s) / (1 + (s-1)·t)`. For 1536 training, 1.12 is steadier than tuning `timestep_shift` alone.
  - At the defaults `timestep_mix_low_prob=0.0` / `timestep_schedule_shift=1.0`, output is bit-for-bit equivalent to the historical sample_t (s==1.0 short-circuits). The InfoNoise CDF phase doesn't read the baseline shift (codified by `test_infonoise_cdf_path_ignores_baseline_timestep_schedule_shift`).
  - PR #85 renamed the field `schedule_shift` → `timestep_schedule_shift` to align with the `timestep_*` family (a zero-cost rename within 24h of merging).
- **The weight bounds for loss_weighting=detail_inv_t can now be tuned in the UI (#72, #87)**
  `loss_weighting=detail_inv_t` previously hardcoded its clamp range to `[1, 5]`; it's now two schema parameters, `detail_inv_t_min=1.0` / `detail_inv_t_max=5.0` (show_when + advanced), whose defaults match the old hardcoded values (no change to existing yaml behavior).

  - Hazy / low-saturation styles: lower max to 3 to ease the "over-learning low t" bias
  - High-contrast / crisp styles: keep the default [1, 5]
  - Aggressive detail: raise max to 8 (with Prodigy, watch the d estimate for single-sample dominance)

  A schema validator fails fast at startup if `min > max`; the lower bound is ≥ 1.0 (the description notes that "<1.0 is ineffective because 1/t>1 always holds" to prevent a dead config zone).
- **Settings adds a "unified model paths" toggle; turning it off lets presets carry absolute paths (#93)**
  On by default (the old behavior, where most users' global model paths sync automatically); when off:

  | | toggle ON (default) | toggle OFF | |---|---|---| | Preset / project page, 4-field UI | disabled + hint linking to Settings | editable + picker + ↺ reset | | fork preset → version | overridden with global values | copies preset values, no override | | save as preset | 4 fields reset to Settings global values | absolute paths kept as-is | | yaml on disk | always absolute POSIX paths | same as above |

  Legacy relative-path yaml is converted to absolute on read/write as a fallback (based on REPO_ROOT, backslashes → POSIX `/`). A dedicated PUT endpoint stays out of the global dirty flow.

### Changed

- **Resuming after a pause continues from the most recent end of epoch, so you don't lose half an epoch of progress (ADR 0006 Addendum 1) (#105)**
  After PR-1 through PR-4 landed, a deep audit of the dev training stack by outside algorithm experts overturned the original ADR's mid-epoch save path — an unguarded grad_accum cycle, a 5% dataloader double-train, the ambiguity of `current_epoch`, and cosine-restart `T_cur` drift were all reasons mid-epoch state can't be frozen. The new approach:

  - **Pause = cancel + immediate GPU release**: handle_interrupt no longer tries to freeze mid-epoch training state; it saves the LoRA as "interrupted" and the child process exits.
  - **End-of-epoch auto-backup**: at the end of each epoch, `state/task_<TID>/auto_epoch_state.pt` is overwritten, so the next resume starts from this most recent complete epoch boundary with no mid-epoch error.
  - **is_pausable in the UI upgraded**: the pause button is hidden entirely during the first epoch / before any auto-backup exists (avoiding the footgun of pausing when there's no resumable state).

  For the full audit process, the outside expert review, and why options A/B/C were rejected, see [docs/adr/0006-queue-pause-resume.md](docs/adr/0006-queue-pause-resume.md) Addendum 1.
- **Running multiple training tasks back to back under the same version no longer overwrites each other's progress files (#97)**
  `save_state_every` / `save_state_every_epochs` / `handle_interrupt` now all write under `output_dir/state/task_<TID>/` instead of the top level of output_dir. Running multiple tasks back to back under the same version no longer overwrites their state files (a latent bug, fixed directly).

  - the supervisor injects the task id via the `LORA_TASK_ID` env var when it starts training
  - running the CLI directly (no env) falls back to a `state/task_unknown/` subdirectory
  - `list_state_ckpts` (the backend for ResumeFieldPicker) scans both the old top level and the new subdirectory, so **existing .pt files from older users aren't lost**
  - Also fixed a small bug where `list_state_ckpts` previously scanned steps but not epochs
- **Exporting the training set / queue now uses native browser downloads, so large files no longer choke on memory (#104)**
  The `<a>` direct-link + backend `*_ready/_failed` SSE pattern from outputs.zip is rolled out to the remaining download entry points (train.zip / queue export), with the browser's native download bar plus a small app-side spinner to finish. In SSH-tunnel / large-file scenarios it no longer eats JS heap, shows progress, and isn't interrupted by switching tabs.

  - Removed `downloadBlob` (fetch→blob, which silently ate memory)
  - New events: `version_train_zip_ready/_failed` / `queue_export_ready/_failed`
  - Preset yaml import semantics are realigned to "upload adds it to the pool and selects it"; a name conflict opens a three-way dialog (overwrite / save as `{name}-2` / cancel) instead of the old two-step flow that switched to create mode, prefilled the form, and waited for you to rename
- **For zip-install users, "enable auto-update" is no longer blocked by local modifications (#84)**
  Observed with v0.8.1 zip users: after unzipping and running `studio.bat run`, npm install changed a few lines of dependency metadata in `studio/web/package-lock.json` → the `git reset --mixed` path left the tree dirty right after init → the pre-flight in the version panel blocked the update. This now uses `--hard` to force the working tree to match the anchor tag. For zip users, "enable auto-update" implicitly means "align with the upstream stable version," so force-overwriting matches expectations better than keeping random changes. The Settings banner copy is changed explicitly to "your local source will sync to upstream v{X.Y.Z}; local modifications in the zip directory will be overwritten."

### Fixed

- **Resuming Prodigy+ScheduleFree / InfoNoise training no longer crashes or loses progress (#105)**
  - `prodigy_plus_schedulefree` (PPSF) relies on `optimizer.train()` to switch into training mode, which wasn't called after `load_training_state`, so resume immediately raised `Not in train mode!`. Fix: `if hasattr(optimizer, 'train'): optimizer.train()` at the end of `load_training_state`.
  - Nine InfoNoise internal states (EMA / K / B / FIFO / cdf / hist / step counters) weren't in the state_dict, so resume cold-started and re-ran N_warm (5000 steps by default). Fix: added `state_dict` / `load_state_dict` serialization, and a K/B mismatch now warns and falls back to a cold start instead of raising.
  - 14 new unit tests codify this (baseline no-op / InfoNoise roundtrip / sample bit-exact / K/B mismatch / FIFO maxlen / corrupt-ckpt warning).
- **Three path-picker fixes: browsing external paths / file start points / Windows path separators (#93)**
  - the endpoint now allows browsing external absolute paths (previously `list_dir` rejected external paths by default, even though PathPicker is designed precisely for choosing external model paths)
  - when a file path is used as the start point, it now falls back to the parent directory and highlights the file (previously a plain 404)
  - the backend outputs `as_posix()` consistently, so the front-end `childPath` concatenation no longer mixes `\` / `/`
  - the model-root input now prefills the actual absolute path as its default; previously the placeholder text wrongly said `REPO_ROOT/anima/` (the actual default is `REPO_ROOT/models/`)
- **LLM tagging now handles more provider response formats and adds concurrency throttling (#88)**
  - parses SSE-style responses (some OpenAI-compatible providers return `text/event-stream` even when stream=false, requiring line-by-line parsing with a `[DONE]` terminator)
  - per-minute concurrent-request throttling (also applies in serial mode)
  - a separate retry for the non-streaming response path
- **A batch of small UI fixes: save-button colors / accidental browser autofill / XY picker applying immediately on selection (#94)**
  - the custom-upscaler download button in Settings was missing the `common.download` zh/en i18n
  - the Settings save button's dirty coloring now toggles between gray `btn-secondary` and orange `btn-primary` (previously it was always orange, with only a weak disabled state to distinguish it)
  - `SensitiveInput` (api_key / user_id, etc.) and the paired `username` now set `autoComplete=new-password` plus `data-lpignore` / `data-1p-ignore` / `data-form-type=other`, preventing Chrome / 1Password / LastPass autofill from triggering a dirty state
  - removed the duplicate "settings" entry for Download / Preprocess from the project step header
  - the Generate XY-axis LoRA picker: `InlineLoraPicker` gains a `live` mode, so chip toggles, weight changes, and pid-vid switches all `onPick` immediately, and the always-present picker in the axis card no longer renders an "add N" commit footer
