Hotfix: decouple the preset system from the redesign + hf-mirror temporarily unavailable + new-preset preview now includes project paths

### Changed

- **Presets are now fully decoupled from versions, with a version's yaml acting directly as its "project-specific config"**
  We found the whole training page's preset mental model was broken:
  - The "Preset X · Customized" label at the top of the picker always showed "Customized", even when you hadn't changed any field — because on fork, default_paths_for_new_version injects the 4 global model paths as absolute paths, which always differ from the relative paths in the global preset yaml. `stripProjectFields` only stripped 6 project-specific fields and missed the 4 model fields.
  - After a fork, the version yaml is actually decoupled from the global preset, but the UI pretended it was still bound, which confused the meaning of "switch preset" and "save".
  - After a fork, `onForkPreset` / `saveNewPreset` didn't call the parent's `reload()` to refresh `activeVersion`, so the label at the top of the picker stayed on the old preset name and you thought the fork hadn't taken effect (the backend had in fact written it).

  Changed to a "version yaml = project-specific config" model:

  - **Picker top button**: when has_config=True it shows "Project-specific config"; when has_config=False it shows "Not configured — pick a preset as a starting point". It no longer shows the source preset name + "· Customized" label.
  - **Picker list cards**: the active highlight is gone (there's no "currently bound" concept anymore), so all cards look the same; the subtitle becomes "Reset to this preset" / "Start from this preset".
  - **Confirm copy**: changed from "Switching presets will overwrite the current version's config" to "Reset to preset X's config? All of the current version's customizations will be lost", with okText changed to "Reset".
  - **Removed the whole baseline-diff logic**: `presetBaselineRef` / `refreshPresetBaseline` / `stripProjectFields` / `customized` are all removed, along with their related useEffect.
  - **Call reload() after fork / saveNewPreset**: to refresh the parent's activeVersion, otherwise the picker top and main form contents fall out of sync with the actual yaml and you'd need an F5 to see it.
  - The `versions.config_name` db field is kept as an audit trail (informational only) and no longer has any UI use.

### Fixed

- **New-preset preview form now includes project paths (reg_data_dir / data_dir / model paths, etc.)**
  The training page's "+ New preset" preview form previously always showed schema defaults (`reg_data_dir=None`, `data_dir=./dataset`, relative model paths), even when the project had already run a reg build and Settings already had models configured. You'd still see a wall of relative paths / blanks in the preview and mistakenly think "forking won't bring the project directories along". In fact `fork_preset_for_version` on the backend **already** injected these values into the version config (verified in practice) — they just weren't visible in the preview, which was misleading UX.

  - **Backend**: `GET /api/projects/{pid}/versions/{vid}/config` adds `project_specific_defaults` in the `has_config=False` branch — merging `project_specific_overrides()` + `default_paths_for_new_version()`, including reg/meta.json detection results.
  - **Frontend startCreatePreset**: uses `project_specific_defaults` to override the schema defaults, so the preview form initializes showing "the values you'll get after a fork" (including reg dir / training dir / output dir / absolute model paths).
  - **Frontend preview form**: matches the main form by attaching `disabledFields` (global model paths grayed out as "Automatic · global settings") + `autoHints` (project-specific fields marked "Automatic · project settings") + `advancedMode`.
  - **Frontend saveNewPreset**: before savePreset it filters out project-specific fields + resets global model paths back to schema defaults, so the global preset pool carries no project / machine data (the backend re-injects on fork, so the version config is still correct). This matches the cleanup logic in `services/presets.py:save_version_config_as_preset`.
- **HF model download default source switched back from hf-mirror.com to the official huggingface.co**
  We found in practice that `hf-mirror.com` fails to download under every tested `huggingface_hub` version (0.25 / 0.30 / 0.34 / 1.14) — the hub client can't read `commit_hash` from the HEAD response and raises `FileMetadataError: Distant resource does not seem to be on huggingface.co`. curl follows the redirect and gets the bytes fine, so the mirror service itself is alive, but the metadata header the hub expects is lost somewhere in the redirect chain — likely a recent change on the mirror's side. This isn't an upstream regression (it was already broken back in the 0.25 requests era), and pinning an old hub would break transformers 5.x.

  - **Default endpoint changed to an empty string** (= the huggingface_hub default = a direct connection to `huggingface.co`): `studio/secrets.py`, Settings.tsx, and the Settings.test.tsx mock are all updated.
  - **Settings UI hides the `hf-mirror.com` preset**: the endpoint field itself still accepts any URL, so you can paste hf-mirror via "Custom URL" to keep trying, or switch to ModelScope.
  - **Recheck doc**: `docs/todo/hf-mirror-recheck.md` records the symptom, root cause, recheck commands, the upstream PR #4071 tracking point, and a reverse-rollback checklist for when it recovers. A recheck every 2 weeks is recommended.
  - The default endpoint field's type / accepted range is unchanged, so users who already configured a custom endpoint are unaffected.
