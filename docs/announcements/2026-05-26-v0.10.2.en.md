Hotfix — LoRA cache pollution when jumping to the Generate page from a project + cross-machine bundle model paths

### Fixed

- **Jumping to the Generate page via "Load in Generate" from a project left stale LoRAs behind, and submit threw axisLoraMissing (#137)**
  The "Load in Generate" button in the completed-state banner on a project's detail view appended the jumped-in LoRA to the end of the locally cached LoRA list instead of replacing it. The result: single-image mode showed the new LoRA alongside old/deleted ones; the left-hand LoRA picker dropdown and chip list stayed stuck on the cached project; and if the cached XY axis was set to "LoRA path" and bound to a LoRA index that no longer existed, clicking "Generate" threw "the LoRA #N bound to the axis doesn't exist." Now a jump is treated as a clear "test this one" intent: it overwrites the cached list with the LoRA named in the URL, the picker remounts and initializes its dropdown with the new project and version, and an out-of-range XY-axis LoRA index collapses to the first entry.
- **Cross-machine bundle import: the source machine's absolute model paths got mangled into a subpath under the local repo (#137)**
  When a bundle exported locally (Windows) included the training config, the 4 global model paths (transformer / vae / text_encoder / t5_tokenizer) were written into the bundle verbatim. On import to the cloud (Linux), those Windows drive-letter paths were read as relative paths and joined onto the local repo root, becoming `<repo>/G:/models/...`, so training couldn't find the checkpoints at launch. The fix aligns with preset-fork behavior: when "auto-sync model paths" is on in Settings (the default), the 4 fields are overwritten with the local global values; when off, the bundle's contents are respected, and cross-platform paths are no longer silently rewritten.
