Full Krea 2 model-family support · fp8 base-model training and inference · VRAM policies and orchestration · unified model-source management · autosaving presets

### Added

- **Krea 2 model family: the full pipeline from training to generation (#408–#425)**
  The training config gains a model-family selector: besides Anima you can now train Krea 2 (a 12.9B single-stream MMDiT with Qwen3-VL text encoding) — LoRA training on the Raw base, samples during training, AI-generated regularization sets, post-training evaluation, and test generation (including 8-step distilled Turbo) all work end to end; output LoRAs interoperate directly with ComfyUI (the Euler + Simple sampling combo matches word for word). See the "Krea 2 support is here" announcement for getting started and training tips.

  - The download center is grouped by family: Krea 2 Raw / Turbo each come in bf16 and official fp8, the text encoder toggles bf16 / fp8; the VAE is shared with Anima, no duplicate download
  - Switching families is a confirmed action: a dialog itemizes every weight path and family default about to be recomputed before anything changes; option dropdowns filter per family, and Anima-only features (NaViT, tag shuffle, etc.) hide automatically on Krea 2
  - Training pre-encodes all captions into a text cache and releases the text encoder by default (saving about 9 GB of VRAM); high-VRAM cloud setups can disable it to keep the encoder resident
- **fp8 base models: training and inference across the board (#426, #434, #435, #436, #437, #444)**
  Official fp8 weights now work as both training and inference base models — pick the fp8 file and it just works, no extra switch. For training (fp8_base), weight VRAM drops 25.6 → 13.1 GB, bringing Krea 2 training down to 24 GB-class GPUs; for inference, resident VRAM is roughly halved, with LoRAs applied via merge-writeback that matches ComfyUI output bit for bit.

  - XY grids support the LoRA-strength axis on fp8 bases (re-merged per cell); LoHa and rs-LoRA load on fp8 bases, DoRA is not supported yet
  - The official fp8 text encoder (5.2 GB) works for both training and generation — one download covers everything; switching precision rebuilds the text cache automatically
- **VRAM and RAM orchestration (#427, #437, #438, #439)**
  Test generation gains a three-level VRAM policy (default·release-after-use / save VRAM / performance): each task pre-encodes all prompts into a cache and releases the text encoder, so it occupies zero VRAM during sampling — a 16 GB GPU can generate with Krea 2 fp8 on the "save VRAM" setting; evaluation and regularization generation use the same orchestration. Loading large weights is now guarded by a RAM check that fails with an actionable error instead of freezing the whole machine in page thrash. "Free VRAM" now actually frees everything (clears leftover cuBLAS workspaces).
- **Community LoRAs from civitai load directly (#426, #428)**
  Krea 2 community LoRAs in PEFT / comfy key format (the common civitai shape) now load directly on the Test page, on both bf16 and fp8 bases, no conversion needed.
- **Unified model-source management (#446)**
  Swapping models for WD14 / CLTagger / eval metrics / upscalers / training base models now uses one shared pattern: a candidate list (built-ins + your additions) with "add download" and "add local file" entries at the bottom; local-file candidates are only ever removed from the list — your files are never deleted. Old configs migrate automatically with selections preserved.
- **One editor for LLM tagger presets (#445)**
  LLM tagger preset editing is consolidated into a single global editor (Settings and the Tagging page open the same one), ending field drift between the two; the Settings LLM section becomes a preset management list (default radio + create / edit), the Tagging page keeps just the per-run preset choice; edits made in Settings take effect on the Tagging page immediately.
- **Generate page: persistent base/TE choice + token count (#441)**
  Your base model and text encoder picks on the Test page are now remembered across pages and sessions (unset = follow global Settings); the prompt box shows a token count; online generation no longer silently truncates prompts beyond 512 tokens.
- **Generation task timeout (#443)**
  An optional task timeout (off by default) force-kills the inference process when the machine is wedged, marks the task failed, and reloads on the next task — no more reboots.

### Changed

- **Default LoRA type for new configs: lokr → lora (#442)**
  New projects and presets default `lora_type` to classic LoRA; existing configs with an explicit value are unaffected.
- **Downloaded assets in the download center now offer "delete" (#437, #442)**
  The implicit overwrite of "re-download" is retired: delete first, then download; tagger / eval / upscaler sections get delete too.
- **Config exclusion rules enforced on both ends + linked-change confirmation (#431, #432)**
  Parameter exclusions declared in the UI are now enforced by the backend as well: invalid combinations in hand-written yaml / CLI fail at startup with a reason instead of passing silently; when a UI change triggers linked field rewrites (e.g. NaViT forcing the attention backend), a dialog lists the changes for confirmation first. The YAML preview is now physically the same serialization as what lands on disk.

### Improved

- **Presets page autosaves + unified save indicator (#433)**
  The global presets page now autosaves like the Train and Settings pages (600ms debounce) — no more save button for existing presets; Train / Presets / Settings share the same "✓ saved" indicator; disabled inputs are now visibly grayed out.

### Fixed

- **Training configs didn't follow a global model re-selection (#439)**
  After picking a different base model / text encoder in Settings, versions with "auto-sync paths" enabled kept showing and using the paths from creation time; reads and enqueued training now follow the global selection.
- **Training task logs showed no step progress (#439)**
  Task logs for Studio-launched training could go a whole epoch without a single step line (output buffering + the progress bar misbehaving outside a terminal); output is now line-by-line in real time.
- **Canceling a task on Windows could leave the training process running (#410)**
  When the process-kill command failed (e.g. access denied), the task showed as canceled while the process kept holding the GPU; termination now falls back to recursively killing the process tree.
