Block swap: train Krea 2 on 12 GB, generate on 8 GB · lower VRAM peaks in test generation · saved images carry Civitai-compatible info · what you see in the training config is what runs

### Added

- **Block swap: train Krea 2 on 12 GB, generate on 8 GB (#463)**
  Later layers of the model stay in system RAM and move onto the GPU only when it is their turn to compute — trading a little time for a lot of VRAM. It works for training and test generation alike: the "block swap" knob lives under System & performance in the training config and under **Settings → Test → VRAM policy** for generation; it is 0 (off) by default. See the "Krea 2 on a small GPU" announcement for how to use it, what number to pick, and what to watch out for.

  - Measured on an RTX 5090 with an fp8 base at 1024²: swapping all 28 blocks puts training at about 10 GB and generation at about 6.3 GB total GPU usage, at about 4% in speed
  - Only where the weights live changes — no training value is affected, and generated images match a run without it
  - Swapped-out blocks stay resident in RAM (about 11 GB for 28 blocks); if you don't have the headroom you are told before the run starts
- **Saved test images carry Civitai / A1111-compatible info (#456)**
  Drop an image into Civitai, A1111 or ComfyUI and the base model, VAE and LoRAs are recognized by name and hash. Only filenames and hashes go into the image — absolute paths from your machine never do.

  - XY grid composites still carry just the Studio parameter snapshot; the full info is written into each cell's own image
- **Pick model paths from what you have downloaded (#461)**
  The base model / text encoder / VAE / tokenizer path fields in the training config gain a "choose model" picker listing the models ready on disk; picking one fills in the path. If there are no candidates it points you to the Settings page, and typing a path or browsing still works as before.
- **Resizable panes on the tag editor and curation pages (#457)**
  Drag the divider between panes to give width to the one you need; the ratio is remembered. Focus a divider and the arrow keys nudge it. Narrow screens still stack vertically.

### Changed

- **What you see in the training config is what runs (#458, #460)**
  A saved version config is never rewritten behind your back — starting a training run leaves the config file byte for byte unchanged. Two behaviors follow from this:

  - Changing the base model in global settings **no longer updates existing versions**. Fields are labeled "default comes from global settings", and when a value differs from the current global one a "restore default" link appears — one click to realign
  - The trigger word field in the training config is retired, and sample prompts no longer get the trigger word prepended automatically. If you relied on it, write the trigger word into the sample prompt itself; the trigger word on the Tagging page is unaffected
- **AI regularization generation defaults to incremental (#453)**
  Switching tags no longer resets the mode back to "full", so a stray click on regenerate can't wipe the images you already have.
- **Consistent image order across step pages (#459)**
  Preprocess / crop / inpaint / tag editor / dataset / regularization now all sort by numeric booru id, matching the curation page. Previously `10.jpg` came before `9.jpg`, and the more images you had the more the pages disagreed.

### Improved

- **Lower VRAM peaks and fewer mid-run failures in test generation (#455)**
  The VAE stays off the GPU until decode time, and when an uncached prompt needs the text encoder back, the DiT steps aside according to your VRAM policy. Measured on 32 GB, the end-to-end peak for a new prompt on the second round drops from 21.2 GB to 17.7 GB.

  - A single image lost to a momentary VRAM shortage is retried once with the same seed instead of failing the whole batch
- **New LoRA merge precision option (#455)**
  On an fp8 base, LoRAs are merged into the weights; **Settings → Test** now controls the precision of that step: fp32 (default) matches ComfyUI, bf16 halves the math so switching LoRAs is faster, at the cost of slight numerical differences. Sampling VRAM is unaffected either way.

### Fixed

- **A custom "saved model name" survives starting a run (#458)**
  It used to look right in the YAML preview and then revert to `{project}_{version}` the moment you hit start.
- **Resume points are no longer cleared when a run is queued (#460)**
  Since v0.20.0, picking a resume LoRA or training state was silently discarded at start, and training began from scratch.
- **PNG metadata records the right prompt and LoRAs on multi-prompt runs (#456)**
  Every image in a batch used to record the first prompt; XY grids also wrote LoRAs that no axis actually referenced.
