Regularization-set editing tools + Generate-page inference aligned with ComfyUI + flip_augment / LoRA loading fixes + logging & error system (ADR 0009)

### Added

- **Regularization sets are now editable: delete images / auto-dedup / dual tagger / mirror+flat / target count (#171, #172, #173)**
  The regularization set has evolved from "once generated, you can only wipe the whole set and rebuild" into an editor that you can edit, dedup, and choose a tagger for, and the generation path now splits into two choices: source + structure.

  - **Delete specific images**: `RegPreview` tabs images by subfolder (`5_concept` / `1_data`, etc.); multi-select delete applies to the current tab. Deleted booru IDs are written to `reg/.deleted_ids.json` and auto-excluded on the next incremental top-up, so deleted images won't be pulled back in.
  - **Automatic dedup loop**: after a build, a built-in dedup loop runs up to 3 rounds (scan for duplicates → keep 1 per group → top up incrementally if short), on by default. The manual "Auto-dedup" button is still there.
  - **Choice of tagging method**: a WD14 / CLTagger dropdown sits next to the auto-tag checkbox (reg sets have a lot of images, so defaulting to LLM / JoyCaption isn't a good fit — that's left for a separate PR).
  - **Mirror and flat structures**: a new `Structure` dropdown. mirror keeps the old behavior, mirroring the train subfolders; flat (the default) uses a single `1_data/` bucket and, together with the "target count" input, pulls by count, decoupled from the number of train images.
  - **AI priors by default + a source picker**: from deep-research findings + a user decision, `generation_method="ai_base"` is now the default (the only path that aligns with the original DreamBooth paper's neutral prior). A radio at the top switches between booru / AI sources, rendering the BooruConfig area or AiGenPanel by source, with a single "Generate" button dispatching either.
  - **find_best_match ranking formula normalized**: the old formula `tag_score + res_score * 0.1` let resolution actually dominate the ranking; the new formula normalizes the tag score first, then uses `0.7 * tag_norm + 0.3 * res_score`, so aspect ratio can really influence which images are picked (the reg set needs to land in the same ARB bucket as train).
  - "Regularization set" renamed "Regularization set (optional)" to make clear it isn't required (modern DiT training doesn't actually depend on a reg set).
  - `docs/user-guide/training-tips.md` gains a Caption strategy chapter, codifying how character / style / character+outfit LoRAs and the reg set work together to get the trigger-as-switch effect.
- **Actual learning rate for PPSF / Prodigy optimizers + a d-value monitor curve (#127)**
  Common Prodigy / PPSF configs set the base lr to 1, so the old monitor chart could easily look like "the learning rate never changes" — even when the actual schedule has already contracted or jumped via `d`. This release surfaces the real learning-rate behavior of PPSF / Prodigy in the Monitor and WandB logs: the derived actual lr, `d`, base lr, effective lr, and the shared group `d` when available. The Studio Monitor gains a d-curve chart.

  It also restores `ppsf_prodigy_steps=0` to PPSF's upstream semantics — `d` is never frozen during training; freezing only kicks in when you explicitly set a step count. The old default froze `d` the wrong way around, systematically underestimating lr for small-dataset LoRAs.
- **Preprocess dedup review gains blurry candidates + a crop / resize relationship report (#161)**
  On top of its existing exact-duplicate / near-duplicate review, the preprocess "dedup review" tool adds two new kinds of quality reports:

  - **Blurry / low-detail candidates**: uses a global Laplacian variance + 12×12 local low-detail regions + the largest connected-region ratio, so it doesn't miss images with locally low detail just by looking at the whole-image average.
  - **Crop / resize relationships**: detects when one image is a crop / local upscale / crop-then-resave of another. It prefilters with a crop-resistant hash + a structure hash, then verifies with grayscale-window matching + local refinement, returning the source image + candidate + window coordinates + match score + relationship type (`crop_smaller` / `crop_upscaled` / `crop_same_area`).

  The parameters area gains advanced thresholds for blur score / local blur / crop score / crop hash / crop edge length, plus performance / recall controls for crop threads / crop tiles / tile coverage / crop aspect ratio / candidates per image. Quality candidates are reports only by default and aren't auto-added to the removal queue; you mark them one by one or batch-select them all at once, and you still have to click "Confirm removal" to write `duplicate_removed` into the manifest.
- **CLTagger gains Copyright / Meta / Quality and other category toggles (5 in all) (#184)**
  The CLTagger model outputs 7 categories (General / Character / Copyright / Meta / Model / Rating / Quality). Previously only Rating / Model had toggles, while the other 5 all implicitly used the general threshold. As a result, LoRA captions would by default mix in tags like `highres` / `lowres` (Meta), `best quality` (Quality), and `fate_series` (Copyright) that you usually don't want for LoRA training.

  Three new checkboxes: `add_copyright_tag` (on by default), `add_meta_tag` (off by default), `add_quality_tag` (off by default), making 5 category gates with the existing rating / model. General / Character are controlled by the threshold and have no checkbox. Both the Tagging page and the global Settings expose this UI in sync.

  After upgrading, default captions no longer include tags like `highres` / `best quality` / `explicit` / `lora_v1`; Copyright (the work's title) is kept, which is the standard LoRA caption shape. To bring any category back, just tick it in Settings or on the Tagging page.
- **Unified logging / error system across backend + frontend (ADR 0009) (#155, #156, #157)**
  Following [ADR 0009](docs/adr/0009-logging-error-system.md), backend + frontend + CLI logging / error handling are brought together onto shared infrastructure. What you'll notice:

  - **Error toasts / ErrorBoundary carry the last 8 digits of a trace ID**: on an API 4xx / 5xx error, the toast ends with `trace ab12cd34`, and a frontend crash ErrorBoundary shows the same trace; a screenshot lets a developer go straight to the matching backend log entry.
  - **Frontend crashes auto-report**: global error capture across three paths — `window.error` / `unhandledrejection` / `ErrorBoundary` — POSTs to `/api/client-errors` (rate-limited 10/min per IP), and the backend writes it to the `studio.client` logger. It also tries to send the moment a tab closes (`fetch keepalive: true`).
  - **Backend trace_id end to end**: every request generates a trace_id in middleware (returned via the X-Trace-Id header), passed across processes via env to the worker subprocess, and written into the JSON lines of `studio_data/logs/studio.log`, so `jq '.trace_id'` can filter every log line for one request.
  - **Unified CLI output**: the 48 `print("[studio] ...")` calls across the `python -m studio` commands are folded into a `_say()` wrapper; the `[studio] X` format you see in the terminal is unchanged, and new output goes through it too.
  - **error_msg written back to the db**: top-level exception tracebacks from the training / tagging and other workers are written into `tasks.error_msg`, so the task detail page shows the failure reason directly instead of just "exit code 1."
- **Creating a preset is now one-click create-and-apply; the two-step draft form is gone (#177)**
  Old flow: clicking "+ New preset" under Train on the project page entered an "inline draft form" state (just a React state flag, zero persistence). After filling fields, switching to another page lost the whole draft — `savePreset` was never called once, and coming back showed "please select a preset" again. The "Create and apply to the current version" button at the bottom was the real save action, but the `noConfigHint` copy made you think clicking "+ New preset" was already creating one.

  New flow: clicking "+ New preset" creates and applies immediately — it auto-generates the name `<project slug>_<version label>`, adds a `_1` / `_2` suffix on name collisions, takes config from the schema defaults (with project paths overriding when "auto-sync model paths" is on), writes to the global preset pool, and forks to the current version. If the version already has a config, an overwrite confirm pops up. The inline draft form and its state are removed, and the UX trap is gone.
- **Native browser upload progress bar + speed + ETA (#139)**
  All three uploads (project local images / zip, training-set bundle import, train zip import) move from `fetch` to `XMLHttpRequest` to get body progress events (which fetch can't provide), showing a progress bar + speed + time remaining. A 1.5s sliding window computes the speed, and once `loaded === total` it switches to a "server processing" state to distinguish upload completion from the unzip / disk-write wait. The bundle import dialog now stays open during upload to show the progress bar, and only closes on success.

### Changed

- **`noise_offset` and pyramid noise change from both-on to a mutually exclusive single choice, aligning with kohya upstream (#175)**
  kohya-ss/sd-scripts PR #477 (2023-05) forbids `noise_offset` and `multires_noise_iterations` from being on together across all models (SD1 / SDXL / SD3 / Flux) — both inject low-frequency content into the noise, and pyramid's lowest-resolution layer ≈ a noise_offset equivalent, so stacking them double-injects low frequencies. Anima's old implementation didn't forbid it, and the normalization at the end of `make_noise` would dilute the constant noise_offset bias added earlier — you'd think both were in effect, when it was effectively "only pyramid running, the offset eaten," which is more insidious than kohya's hard error.

  Adds a `noise_enhancement_type: "none" | "offset" | "pyramid"` field (a frontend dropdown); the mutually exclusive opposing fields auto-hide and are forced to zero. Old yaml migrates automatically: when both are > 0, pyramid takes priority (pyramid was the one actually in effect in the old implementation), and the offset field is zeroed.
- **Settings becomes a right-slide drawer, available globally + without interrupting the current page (#132, #133)**
  The Settings page (formerly the `/tools/settings` route) becomes a right-side slide-out drawer, width `max(1024px, 70vw)`, dismissible via backdrop / ESC / the × in the title bar. The 8 entry points (sidebar "Settings" / Topbar / CommandPalette / Train / Presets / Tagging at 4 spots) now open the drawer instead of navigating a route, and the old URL is handled by a compatibility redirect. The sidebar stays clickable while the drawer is open (switching project / version / theme doesn't require closing it first), and closing the drawer in a dirty state pops a confirm.

  The data layer is lifted to a root-level Provider (secrets / catalog / SSE persistence), so mounting / unmounting the drawer no longer re-fetches secrets. Settings is split into its own `React.lazy` chunk (~116KB / 27KB gzipped), kept out of the first-screen bundle.
- **Task archives moved to tasks/id/ — multiple tasks in the same version no longer cross-contaminate + deleting a version doesn't lose history (#166, #192)**
  A training task's monitor state / sample images / run log move from `versions/<v>/monitor/task_<id>/` to `studio_data/tasks/<id>/`, sharing a root with the task config snapshot:

  ```
  tasks/<id>/snapshot/config.yaml    ← the config frozen at task launch
  tasks/<id>/monitor/state.json      ← loss / LR / sample index
  tasks/<id>/samples/*.png           ← training sample images
  tasks/<id>/run.log                 ← worker stdout/stderr
  ```

  - Deleting a version no longer loses historical tasks: to look back at "the loss curve / sample images from the last run of this config," you no longer need to keep the corresponding version around.
  - Running several tasks back to back under one version no longer cross-contaminates: in the old implementation the second task's monitor state would overwrite the first, so clicking "View" on task1 showed task2 — fixed in this release.
  - Staying at the version level: `reg/` (reused across tasks), `output/*.safetensors` (the version's deliverable), `config.yaml`, `train/`, `caption_snapshots/`.
  - Old-task compatibility: the DB `monitor_state_path` column keeps its old value untouched, the sample / log readers keep a fallback candidate for the old paths, and pre-PP6.1 (< v0.5.0) historical tasks are still readable.
- **Full review of schema field descriptions + fixed 4 backwards descriptions + `noise_schedule` split into 3 groups (#194)**
  - **4 backwards descriptions corrected**: `ppsf_d_coef` / `prodigy_d_coef` used to say "overfitting → 2.0," which is backwards — you raise it when underfitting and lower it when overfitting; `huber_c` had the δ direction backwards (the larger δ is, the closer to MSE; the smaller, the more tolerant of outliers); `min_snr_gamma` suppresses the high-SNR easy steps at low t (under rectified flow, SNR and t run in opposite directions). Anyone tuning by the old descriptions had the direction reversed.
  - **`noise_schedule` group split into 3**: the original 22 items actually mixed 3 concerns, split into `noise_augmentation` (4 items, noise_offset / pyramid) / `timestep_sampling` (11 items, timestep + InfoNoise) / `loss` (7 items, huber / min_snr / detail_inv_t). The frontend config page's group labels change to match, 3 groups.
  - **Wording cleanup**: removed dev-notes (internal references like "kohya upstream forbids this too"), removed PR-author subjective experience (paper-unsupported assertions like "for hazy art styles, try 3"), tunable numeric parameters consistently note "what larger / smaller does," and unbounded fields get a typical-range warning.
- **"Global proxy" renamed "Booru proxy" + moved to the Dataset tab + removed the standalone Network tab (#191)**
  The "Global proxy" section added in v0.10.3 was called "Global proxy" in the UI, but the actual data flow only feeds the booru download path (neither ModelScope nor HuggingFace downloads go through it). The original version had a chunk of code trying to push the proxy into the hf_hub httpx client that imported a symbol that doesn't exist in hf_hub; after hotfix #165 removed it, only the booru path was left using it.

  UI changes: the section is renamed "Booru proxy," moved to the end of the Dataset tab (same tab as Gelbooru / Danbooru), the standalone "Network" tab is removed, and the copy makes clear "covers only the Booru download path; model downloads don't use this proxy." The `secrets.proxy` schema is untouched, so your saved config isn't lost, and a stale `activeTab="network"` left in old localStorage auto-degrades to the default tab. Routing model downloads through a proxy still needs OS-level `HTTP_PROXY` / `HTTPS_PROXY` environment variables or an `HF_ENDPOINT` mirror.

### Improved

- **Monitor page charts / sample images / config field order / tagging blacklist input fixes (#166, #186, #187, #189)**
  A batch of UI fixes on the Monitor and Tagging pages, converging over several iterations:

  - **Charts handle both large and small screens**: the three charts in the right column now use `useLayoutEffect` + `ResizeObserver` to measure real pixel dimensions and render the SVG 1:1, so text / line width / dots aren't distorted by the parent's aspect ratio; the x/y axis font goes 9 → 13 and padding increases so a 6-character-wide y-label like `0.0796` doesn't overflow; the first / last x-tick switch to `start` / `end` anchors so edge labels don't spill halfway off; the three right cards are clamped to `[140, 300]px` to avoid 4K proportion imbalance; and 1080p no longer gets a stray scrollbar.
  - **The d curve becomes its own third card**: previously crammed inside LR, its 50px height was squashed; as its own card the height aligns with loss / LR.
  - **Smooth sliders separated**: loss defaults to 0.02, LR / d default to off (LR / d are already EMA-derived, so stacking another EMA distorts them).
  - **Click to enlarge sample images**: the large sample images on the Monitor page didn't respond to clicks before; now they get `cursor-zoom-in` + fullscreen preview + ← / → to move through the sample sequence.
  - **Sample image cards no longer blown out by the source image**: the large sample `img` switches to `absolute inset-0` to leave the in-flow, so the browser doesn't treat the 1024×* source image's intrinsic size as the parent's min-content and push the sample card up to the source's height.
  - **VersionRail syncs the sidebar when switching versions**: switching versions in the project overview's VersionRail missed calling `ctx.onSelectVersion`, so the sidebar's phase nav didn't follow along; one added call fixes it.
  - **Tagging blacklist input couldn't type commas / spaces**: the controlled input split → trim → filter the text back into an array on every keystroke, so the empty segment a comma produces got filtered out and a trailing space got trimmed, so you couldn't even type `blue eyes`. Extracted a `TagsInput` component: plain text while editing (focused), rendered into chips on blur. Synced across both the Tagging page and the global Settings.

### Fixed

- **Generate-page inference aligned with ComfyUI output + LoRA DoRA / rs_lora no longer silently fails (#169)**
  Users reported two separate symptoms: on the Generate page "the LoRA had no effect at all," and the same config produced clearly different output in Studio vs. ComfyUI. Going through the native ComfyUI Anima implementation item by item confirmed 4 alignment bugs at different levels.

  - **LoRA DoRA / rs_lora pass-through**: `read_lora_meta` missed reading the `weight_decompose` (DoRA) and `rs_lora` fields from `ss_network_args`, so `apply_loras` built a network structure inconsistent with training — all 280 of DoRA's `dora_scale` tensors were skipped as unexpected, and rs_lora's α scaling was cut to 1/8 of what training used. The two together are what you perceive as "the LoRA had no effect at all."
  - **T5 token weights take effect**: T5 token weight syntax like `(tag:1.3)` was computed but then discarded by `preprocess_text_embeds`, so the weights had no effect; now, following ComfyUI, the t5xxl_weights are multiplied in after the LLMAdapter output.
  - **Initial noise reproducible across hardware**: `torch.randn(device='cuda')` goes through cuRAND (inconsistent across hardware); switched to a CPU generator + `.to(device)`, so the same seed is reproducible across platforms.
  - **Default negative prompt**: when you leave it blank, it no longer sneaks in a long `"worst quality, low quality, score_1, ..."` string, so the CFG anchor matches ComfyUI.

  After comparing the native ComfyUI Anima implementation file by file in parallel with several agents (VAE / DiT 685 keys / RoPE 3D / Sampler / Scheduler / CFG) and confirming the rest of the path is fully equivalent, this release changes only these 4 spots.
- **cache_latents and flip_augment on together permanently mirror-pollutes 50% of the data (#176)**
  During its cache stage, `CachedLatentDataset` calls `ImageDataset.__getitem__` to get pixels, but the base dataset's `if flip_augment and random.random() > 0.5: img.transpose(...)` rolls the dice during the cache stage — every image is permanently encoded as either flipped or not, and training afterward only reads the npz and never invokes the base flip. The consequence: flip_augment silently does nothing (you think you have data augmentation, but you don't), and 50% of the data is permanently mirror-polluted (a random half of the images get mirrored at each training launch).

  Fixed following kohya-ss/sd-scripts' `ImageInfo.latents / latents_flipped` dual-latent approach: the cache stage encodes single / double copies depending on `flip_augment` (the double stores both `latent` + `latent_flipped` keys in the npz), and at training time `__getitem__` picks which key by `random.random() > 0.5`. The cost: with `flip_augment=True`, both the encode time and cache size are ×2, but flip really takes effect in training.

  Old caches self-heal: `_is_cache_valid` invalidates and re-encodes when `flip_augment=True` but the npz is missing the `latent_flipped` key; conversely, with `flip_augment=False` the double copy is still valid (a superset), so toggling the switch doesn't re-encode back and forth.
- **Generate-page XY grid mixed in LoRAs that were never selected (orphan anchor buildup, since v0.8.0) (#167)**
  In XY mode users kept seeing "LoRAs they never selected" mixed in: the XY axis had chenbin V3.4 selected, yet every cell got pinned with an unselected chen-bin v3.2 or a cross-mode hoshi. It reproduced even after clearing localStorage / switching to an anonymous browser, unrelated to the frontend cache.

  Root cause (present since the v0.8.0 XY feature): the only entry point that adds an anchor to the LoRA bucket in XY mode is the `lora_ckpt` axis picker, but it "only pushes, never prunes" — switching project / switching axis type / deleting the Y axis only clears the axis binding, leaving the bucket's anchors untouched. The XY sidebar has no base-LoRA list UI, so you can't see or delete them. At submit, the old logic sent the whole bucket as base LoRAs, and the backend stacked them onto every cell, so the orphan LoRAs appeared constantly in every image. Past fixes (including the recent dual-bucket split) only addressed cross-mode bleed and never touched the in-bucket orphan buildup + sending the whole bucket as base.

  Fix: extracted a pure function `buildXYMatrix` that, at submit, keeps only the anchors referenced by the X / Y axis `loraIndex`, remaps the indices in order of appearance, and discards orphans; `lora_configs` is empty for non-lora_ckpt axes.
- **Generate-page enqueue race (`config not found`) + LoRA bleed across single / xy modes (#159)**
  - **Enqueue race**: `enqueue_generate` first writes the task as `pending+generate` (which makes it dispatchable), then builds the config (including the time-consuming `detect_attention_backend`) and writes `config_path`. The supervisor ticks every 1s; if it dispatches in the window where "`task_type` is already generate but `config_path` is still NULL," the daemon reads an empty path and goes straight to failed. Changed so `_dispatch_generate` skips tasks whose `config_path` isn't in the db yet (dispatched on the next tick), and marks a task failed when config build fails to avoid it staying pending forever.
  - **LoRA bleed across single / xy modes**: `loras` was persisted as a single shared array, so adding A in single, switching to xy and adding B, made xy generate with A+B, and switching back to single showed A+B too. Split into `singleLoras` / `xyLoras` dual buckets routed by mode (compare shares the xy bucket), with old config auto-migrated into one copy in each bucket so selections aren't lost.
- **XY lora_ckpt axis ordered by the ckpt display order, no longer jumping around by click order (#164)**
  On the Generate page's XY grid, the `lora_ckpt` axis (sweeping the same LoRA's different epoch / step checkpoints to find the overfitting inflection point), when multiple ckpts are selected, ordered the axis values by the order you clicked the chips, not by epoch / step. The consequence: picking ep80 / ep60 / ep40 in a jumbled click order (80 → 40 → 60) laid out the grid columns as `[80, 40, 60]`, dropping ep60 at the end so the eye couldn't follow the "more overfitting the later you train" trend and the inflection point couldn't be read.

  Root cause: `InlineLoraPicker` stores the multi-select in a `Set<string>`, and at commit `Array.from(set)` takes the Set insertion order = click order. Changed to the canonical sort already computed by the backend's `list_lora_ckpts` (`final → step↓ → epoch↓ → other natural order`, which the picker already renders the chips in).
- **`sample_seed=0` resolved at training start, same prompt and noise across epochs (#193)**
  The semantics of `sample_seed=0` is "random seed" (stated in the schema description), but the old implementation's `if s_seed:` skipped the set-seed entirely and relied on the global torch RNG throughout. The global RNG is consumed by dropout / timestep / noise / dataloader at every training step, so its state is completely different by the time each epoch reaches the sampling point — the same prompt produces a different image at different epochs. Sample images across epochs are meant to show "the convergence process / overfitting trend," which assumes the noise is fixed so only the model changes; the old behavior changed the noise too, ruining the comparison.

  Following kohya-ss/sd-scripts' standard approach: during the training bootstrap, if `sample_seed` is 0 / unset, draw a random seed once, write it back to `args.sample_seed`, and log it, so the whole run uses the same prompt and same seed. pause / resume freezes through the snapshot, so it uses the same seed across pauses; if you restart a task while the yaml is still 0, it draws a new random one.
- **Windows launch reports npm WinError 193 (npm resolution prefers `.cmd`) (#140)**
  Launching `python -m studio` on Windows reported `OSError: [WinError 193] %1 is not a valid Win32 application`, with the stack hitting `subprocess.Popen([npm, ...])`. Root cause: `find_npm`'s candidate order let `shutil.which("npm")` match `C:\Program Files\nodejs\npm` first — that's the bash script (no extension) the official Node.js installer ships for Git Bash / MSYS, which `CreateProcess` can't launch directly.

  Fix: the Windows candidates are reordered to prefer `.cmd` → `.ps1` → bare name as fallback; on a `.ps1` hit it auto-wraps with `powershell.exe -NoProfile -ExecutionPolicy Bypass -File`. Linux / Mac are narrowed to `("npm",)` by `os.name`, so their behavior is completely unchanged.
- **`studio.sh` now pauses on a launch error instead of flashing closed (so double-click users can see the error) (#174)**
  When Node.js isn't installed, `studio.sh` only printed a single `Exit code 2` line and exited, so users launching from the file manager or a non-persistent terminal saw the error flash by and disappear. `studio.bat` has long had a `pause >nul` backstop; this release gives `studio.sh` the matching handling — a nonzero rc + a TTY prompts to press Enter before exiting; a normal exit / Ctrl+C / non-TTY (CI / pipe) is skipped to avoid a false hang.
- **Missing SOCKS5 proxy dependency caused base-model download to report `socksio not installed` (#190)**
  After configuring a `socks5://...` proxy, downloading the base model reported `Using SOCKS proxy, but the 'socksio' package is not installed`. Root cause: `requirements.txt` didn't declare the extras SOCKS5 needs. The two download paths use different HTTP clients, each needing a different SOCKS backend — HuggingFace (httpx) needs `socksio`, booru / ModelScope (requests) needs `PySocks`. `requirements.txt` is changed to `requests[socks]>=2.31.0` + an explicit `httpx[socks]>=0.27`, with each extra serving its own path, and no C extensions so it's cross-platform fine.
