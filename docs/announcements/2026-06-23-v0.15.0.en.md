Multi-resolution training + CLTagger V2 + custom base models + a movable model root directory + Settings redesign + localized error messages + various self-update / ONNX fixes

### Added

- **Multi-resolution training: mix several base resolutions in one run + configurable bucket aspect ratios + a bucket-distribution preview (#296)**
  Official reference LoRAs often mix several resolutions (e.g. 512/768/1024) to improve high-resolution inference; until now the trainer only supported a single base resolution, and works with extreme aspect ratios were hard-cropped by a fixed 2:1 bucket. This version adds multi-resolution training while keeping the existing aspect-ratio bucketing.

  - The training resolution can take multiple values (e.g. `512, 768, 1024`); each image in a folder with no prefix is trained once at each resolution.
  - Adding a `1024px_` prefix to a folder name (e.g. `1024px_2_data`) pins that folder's resolution, overriding the global list.
  - New `aspect_ratio_limit` (default 2.0): raise it to let tall images bucket at their original ratio instead of being hard-cropped.
  - The right column of the training page shows the actual bucket distribution (size and effective image count per bucket); buckets that can't fill a batch are flagged in red (the whole bucket is dropped).
  - Also fixed: setting the base resolution to 512 previously left only a single square bucket (bucketing broke and every image was squashed to square); buckets are now generated correctly from the base resolution.
- **New CLTagger V2 tagger (gated repo + external weights; V1 behavior unchanged) (#297)**
  The CLTagger tagger gains support for a V2 version (`cella110n/cl_tagger_v2`). V2's weights and tag-vocabulary structure differ from V1 (a 2GB weights file kept externally, a separate vocabulary, RGB preprocessing).

  - V2 is a gated repo; downloads automatically include the HuggingFace token you entered on the Secrets tab, with an actionable message when the token is missing or unauthorized.
  - The external weights file is included in the integrity check, so a half-finished download won't be misjudged "ready" and then fail opaquely at load time.
  - V1 (`cl_tagger_1_02`) behaves exactly as before and can keep being used; switch versions in the tagging section of Settings.
- **Register local fine-tuned weights as a base model + temporarily swap the base model for AI-prior and test image generation (#302, #309)**
  Until now the base model could only be chosen from the few official versions; there was no first-party entry point for generating test images on your own fine-tuned weights or using a locally downloaded `.safetensors` as the base model. This version adds it.

  - Settings → Training → the Anima base model now offers "add local model," registering an existing `.safetensors` on disk as a base model (it only records the path — nothing is copied or deleted).
  - The regularization-set AI-prior generation and the Generate page each gain a "base model" dropdown for temporarily switching to another downloaded version (including local fine-tuned weights) without touching the global setting; the dropdown lists only models that actually exist on disk.
  - Also fixed: the test / Generate page wasn't actually following the base model version chosen in Settings (it was hard-wired to one version); it now follows it for real, so testing on fine-tuned weights takes effect.
- **The model root directory can move to another drive + a one-click migration of already-downloaded weights (no restart needed) (#304)**
  Downloaded model weights default to `models/` under the repo and keep growing. Just like the studio_data directory, you can now move the model root directory to another drive and copy the already-downloaded weights over in one click.

  - Settings → System → Storage location adds a "model root directory" card; change the location → confirm the scan (file count / total size) → copy with a progress bar.
  - It takes effect immediately, with no restart (unlike the studio_data migration); the old directory is left intact, and old absolute paths hard-wired into old training configs still resolve.
  - Migration is refused while training, tagging, or other tasks are running, to avoid files being truncated mid-write during the move.

### Changed

- **A Secrets tab in Settings centralizes credentials, and the download source is now chosen per model type (#301)**
  Credentials and download sources in Settings have been reorganized.

  - A new "Secrets" tab centralizes HuggingFace / ModelScope / Gelbooru / Danbooru credentials (previously scattered across the training / dataset / monitor tabs — for example, the HuggingFace token CLTagger needs had to be found on the training page).
  - The download source changes from a single global HF/ModelScope toggle to a per-type choice on each model card (a separate dropdown for the training model / WD14 / upscaler); the old global setting migrates automatically as the initial value rather than silently reverting.
  - Error messages on download failure are now more actionable (e.g. a gated repo missing a token prompts you to enter one directly), instead of only being visible in the terminal.
  - A few fields that were filed under Gelbooru but actually shared by all downloads / uploads (keep tags, convert to PNG, strip the alpha channel) move under the global download settings, with old values migrated automatically and nothing lost.

### Improved

- **Error messages are localized to the app language (Chinese / English); the ADR 0009 error-envelope migration is complete (#294)**
  The app was already bilingual, but error toasts were hard-coded in Chinese, so English users saw Chinese error messages. As of this version, error messages are localized to the app language.

  - About 120 error codes are covered in both Chinese and English; error messages for cases like download failures are also more specific and actionable (e.g. a gated repo missing a token prompts you to enter one directly).
  - Every backend error response now carries a semantic error code + a trace ID, and the front end looks up the localized message by error code; codes not yet covered fall back to showing the original English message rather than going blank.
  - With this, all three phases of the ADR 0009 error-envelope migration (unified envelope → error-code localization → cleanup of old fields) are complete.
- **Settings field titles / layout / toggle styles unified + redundant options removed (#306)**
  Settings got a round of UI cleanup to unify inconsistencies accumulated over time.

  - Field titles are now plain language (no longer the raw config field name — e.g. `exclude_tags` shows "Excluded Tags," `api_key` shows "API Key").
  - Boolean toggles are unified into an enable/disable dropdown (fixing vertical alignment); form-control widths and styles are unified, and the content area's max width is widened to suit wide screens.
  - The System page's "rerun first-run setup" is folded into the "Services" section; the storage / model-root paths change from code blocks to selectable, copyable read-only inputs.
  - The tagger's "manually specify a local model directory" option is removed (the model root directory can already set the landing location uniformly, making it redundant); leftover values in old configs are dropped automatically on load.
- **Training-monitor sample thumbnails show epoch + step in their badge + the broken standalone monitor button is removed (#307)**
  - The sample thumbnails / full images on the training monitor page now show epoch + step in their badge (epoch was previously hidden only in the file name, so you had to open the full image and read the file name to know which round it was).
  - The thumbnail badge moves from overlaying the 64px thumbnail (covering the image) to a compact line below it, keeping the image itself clean.
  - Removes the long-broken "standalone monitor" popout button (it was missing the app path prefix, so clicking it landed on an error page); the sidebar "Monitor" entry already includes task selection, making this button redundant.

### Fixed

- **Fixed onnxruntime-gpu installing the CUDA 13 build by mistake on Linux, crashing tagging (#311)**
  After installing the ONNX CUDA build, Linux users saw tagging fail with `libcudart.so.13` not found and a misleading "onnxruntime not installed" — the package was actually installed, it just failed to load.

  - Root cause: the default PyPI wheel for newer onnxruntime-gpu (from 1.27 on) switched to CUDA 13, while this project is built end-to-end for CUDA 12 (torch is cu128, the preloaded libraries are all `.so.12`), and mixing them crashes at load time.
  - This version installs the matching onnxruntime-gpu build based on torch's CUDA major version, pinning CUDA 12 to the 1.20–1.26 line; when torch later moves to CUDA 13, it will switch to the corresponding build automatically.
  - The Settings → ONNX Runtime info box adds a line showing torch's current CUDA major version, with a note when there's a mismatch.
  - Users who already have a broken install can run `pip install "onnxruntime-gpu==1.20.1"` and restart to recover immediately; the DirectML / CPU / Windows paths are unaffected.
- **Self-update hardening: npm rewriting the lockfile no longer blocks upgrades + upgrading from old versions no longer re-downloads model data (#308, #310)**
  Two kinds of reliability issues in the WebUI self-update (`git fetch` + `git reset --hard`) are fixed.

  - Newer npm automatically appends metadata like `peer` to `package-lock.json` at startup, making this version-controlled file "dirty" out of nowhere and stalling the "working tree must be clean" pre-update check, so the update button wouldn't respond. As of this version, this kind of automatic rewrite no longer counts as dirty or blocks updates.
  - If the working tree really does have manual changes, a "force overwrite and update" confirmation appears before updating (shared with rollback), so a reset won't silently overwrite your real changes.
  - Early versions had mistakenly committed a few downloaded model data files (tokenizer / config, a dozen-odd MB) to git and later moved them to gitignore; upgrading from a version of that era, `reset --hard` would delete them and force a re-download. This version backs up and restores those files around the upgrade, so old users won't notice (the large weights files were never committed and are unaffected).
- **Fixed grad_accum dropping the last partial group's gradients on small datasets (one fewer step per round) (#305)**
  When a dataset's per-round batch count isn't divisible by the gradient-accumulation step count (`grad_accum`), the gradients of the last under-filled group of batches were discarded and not counted toward training.

  - Extreme case: with 3 images / `grad_accum=4`, an entire round had no update step at all and didn't train.
  - Across multiple rounds, the last group's gradients also leaked into the next round and were normalized incorrectly.
  - As of this version, an under-filled group at the end of each round still steps normally and is normalized by the actual batch count (so the last step's gradient isn't weakened). Configs that already divide evenly, or use `grad_accum=1`, behave the same as before.
- **Fixed a torch.stack crash when training with caching off + batch>1 + non-square images (#299)**
  With latent caching off (`cache_latents=False`) + a batch larger than 1 + a training set whose aspect ratios aren't all the same, training crashed outright with `stack expects each tensor to be equal size`.

  - This path was missing aspect-ratio bucketing, so a batch mixed images of different sizes and crashed at concatenation.
  - The cached path (on by default) already avoided this; this version makes the non-cached path batch by bucket too. Multi-resolution training (new in this version) is especially prone to triggering it with caching off, and it's fixed along with the rest.
