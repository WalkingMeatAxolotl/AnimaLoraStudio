Hotfix for Danbooru search-API 403s after it moved behind Cloudflare

### Improved

- **Danbooru now requires an account; anonymous access is no longer supported (the UA also carries a by-username identifier)**
  - The UA includes `(by username)`: this matches Danbooru's recommended TOS format, and when Cloudflare tightens up, an account allowlist is safer than an anonymous UA
  - `secrets.has_credentials_for("danbooru")` now checks for `username + api_key`, consistent with gelbooru
  - The old comment claiming "anonymous works too" is no longer true (in the Cloudflare era, anonymous gets nothing through), so it's now an explicit requirement
  - The Settings placeholder now reads "Required — Danbooru no longer supports anonymous access since moving behind Cloudflare"

### Fixed

- **Danbooru search API 403 — an app User-Agent gets it past Cloudflare**
  After Danbooru moved behind Cloudflare, the search API returned 403 across the board (the `Just a moment...` challenge page).

  - `services/booru_api.search_posts` previously set no `headers`, so requests' default UA `python-requests/X.Y.Z` was blocked outright by Cloudflare
  - It now uses the app-name UA `AnimaLoraStudio/0.5.2` rather than spoofing a browser (in testing a Chrome UA still got 403s — Cloudflare treats it as a "browser that doesn't run JS" scraper)
  - Added `Accept: application/json` to make middleware routing more deterministic
  - `pynvml` → `nvidia-ml-py` (already added in PR #37, unified in this release)
