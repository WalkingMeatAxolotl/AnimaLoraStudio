Project / version / task lifecycle overhaul (ADR 0007) + a multi-tool preprocessing flow + pre-training dedup review

### Added

- **Task detail gains an "Associated config" tab + one-click reuse of that config (#117, #119)**
  When a task starts, its training config is frozen to `studio_data/tasks/{tid}/snapshot/config.yaml`, kept mentally separate from the version's current config.

  - The new "Associated config" tab on the task detail page shows the full yaml the task ran with (read-only).
  - An "Apply this config" button writes the snapshot back to a new or existing version, making it easy to reproduce past training parameters.
  - Later edits to the version / preset / yaml don't affect a historical task's snapshot, so you can read back the exact parameters it ran with across devices and platforms.
- **Pre-training dedup review tool: review groups by hand to keep or remove (#111, #126)**
  The preprocessing toolbar adds "Dedup review" (route `/preprocess?tool=dedupe`). It scans the current working set, shows similar / near-duplicate images grouped together, and after you review each one it only writes a `duplicate_removed` skip status to the manifest — it never moves or deletes the originals in `download/`.

  - A dedicated OperationPanel at the top: common options (match scope / difference score / threads) are surfaced; advanced algorithm parameters (hash side length / structural threshold / aspect-ratio tolerance / tiling grid) are collapsed by default.
  - The workspace shows each group's thumbnails with a suggested-keep badge and per-image match type / score / dimensions / file size.
  - A free two-state choice (green = keep / yellow = remove), toggled independently per image, with no minimum on how many you must keep in a group.
  - A "Select suggested" button applies the system's suggestions in one click.
  - The confirmation dialog requires that you've reviewed the actual group images.
  - Adds the dependency `ImageHash >= 4.3.0`.
- **One image, multiple crop boxes fan out into separate derived images + cluster cropping auto-aligns to training resolution (#112)**
  - **One image, multiple boxes fan-out**: on the crop page a single image can have multiple rectangles drawn; on confirm they split into separate derivatives `X_c0.png` / `X_c1.png`. Filtering / `copy_to_train` / thumbnails all address the derivative names end to end, and the original no longer takes its own slot.
  - **Cluster cropping aligned to training buckets**: a new `lib/trainBuckets.ts` 1:1 mirrors `runtime/training/dataset.py:BucketManager` (37 buckets with a strict sync assertion); clustering snaps internally to a training bucket's aspect ratio so crops land exactly on a training bucket and the trainer doesn't resize a second time. UI labels still use readable aspect ratios (4:3 / 16:9).
- **Dataset bundle export / import: download locally or pick a server path (#108)**
  A new flow for packaging and exporting a dataset:

  - Export can download locally or export to the project directory.
  - The import dialog supports browser upload and server path selection.
  - The old standalone `data_exports` import entry is removed (server paths cover that case).
  - The dataset page's old "Download dataset" button is removed (folded into bundle export).
- **Per-layer rank via `lora_rank_rules`: assign different ranks by layer-name regex (#108)**
  The training config adds an optional dictionary field `lora_rank_rules`, keyed by a layer-name regex with a rank value, e.g.:

  ```yaml
  lora_rank_rules:
    "lora_unet_.*double.*": 16
    "lora_unet_.*single.*": 8
  ```

  Layers that don't match any rule fall back to the global `lora_rank`. This makes it easy to allocate the parameter budget differently by module importance.
- **Tag editing gains tabs to switch between folders (#108, #113)**
  The tag editing page splits into per-folder tabs at the top, so a multi-folder project can jump to any folder's view quickly, with each tab counting independently.
- **Preprocessing overview gets three tabs: All images / Processed / Removed (#126)**
  The preprocessing overview becomes a three-tab view:

  - **All images**: the current working set (excluding removed), read-only browsing with click-to-zoom.
  - **Processed**: preprocess derivatives, with the existing select-to-restore and undo-all.
  - **Removed**: images soft-deleted by dedup, with visible thumbnails (sourced from download/source) that you can restore with one click after selecting.
- **Click-to-zoom on the "Processed" tab becomes a side-by-side before/after comparison (#126)**
  The preview modal adds a `compareSrc` mode: when provided, it splits left/right into a two-image comparison (left = the download original, right = the preprocess derivative), making it easy to see how a multi-crop derivative maps to its original. On narrow screens (< md) it stacks vertically. Without `compareSrc` the behavior is unchanged.
- **The Generate page gains a live daemon log drawer (#128)**
  The Generate page adds a "Logs" button (next to "Clear VRAM"); it slides a drawer up from the bottom to 40vh showing the inference daemon's live logs.

  - The terminal no longer prints daemon output by default — it all goes through the UI drawer.
  - The backend `InferenceDaemon` adds a ring buffer (deque maxlen=2000) + a log listener, and the supervisor relays incremental `daemon_log_line` SSE pushes.
  - A new endpoint `GET /api/generate/daemon/logs?since_seq=N` pulls historical logs.
  - When hidden the drawer uses `translateY(100%)` + `pointerEvents:none`, so it takes no layout space at all.

### Changed

- **Project / version / task state model upgrade + sidebar drives a cursor forward (#115, #117, #119)**
  The project–version–task state model goes from a single `stage` field to two fields, `status` and `phase`. `status` has 5 states: preparing / training / completed / failed / canceled; `phase` is a 5-state cursor: curating / tagging / editing / regularizing / ready.

  - The sidebar becomes the main channel for advancing the cursor: expanding a project shows the phase list with progress, and the cursor+1 row is clickable to advance / skip directly. The standalone PhaseHeaderNav is removed.
  - Completed phase numbers turn green; the current phase is highlighted with a red number; phases past the cursor are disabled.
  - DB upgrade v9 destructively drops the `projects.stage` / `versions.stage` columns (recreate-table mode for SQLite < 3.35 compatibility).
  - Full design / decisions / migration strategy in [ADR 0007](docs/adr/0007-project-version-task-lifecycle.md).
- **Project detail page v2 redesign (StatusBanner / VersionRail / dataset thumbnail cards) (#121)**
  The project detail page is rebuilt to design v2:

  - An identity strip + VersionRail (horizontal version switching) + a 5-state StatusBanner (preparing / training / completed / failed / canceled, each with its own CTA) + an asymmetric 2+3 detail grid.
  - The dataset card uses ImageGrid + ImagePreviewModal + folder chips, matching the "Dataset filter" panel.
  - The tag distribution card embeds a progress bar, with trigger words marked by ★ and accent-highlighted.
  - The pixel distribution card reuses BarHistogram, sharing the same data as the zoom page's sidebar.
  - For completed projects, the LoRA files tab reuses the queue detail page's OutputsTab, with download / package / open folder.
  - All CTAs hook into existing APIs: fork version → opens NewVersionDialog prefilled; load LoRA → jumps to the Generate page prefilled; view logs → jumps to queue detail #log.
- **Preprocessing restructured into multiple tools: overview / upscale / crop / inpaint (#112)**
  Preprocessing evolves from a single zoom page into a multi-tool structure at route `/projects/:pid/preprocess?tool=overview|upscale|crop|inpaint`.

  - Toolbar: overview / dedup review / upscale / crop / inpaint, sharing one outer frame.
  - Overview page: a grid of all preprocess workspace images + ctrl/shift multi-select + single-image preview modal + undo selected / undo all.
  - Upscale page: filter chips become pixel bins (< 512² / 512²–768² / 768²–1024² / 1024²–1536² / 1536²–2048² / > 2048²), sharing the sidebar's pixel distribution; the JobStrip hides entirely when there's no live job.
  - Crop page: the filmstrip moves from a bottom horizontal row to a left vertical 3-col square cover-thumb strip; FreeCropEditor adapts with a ResizeObserver.
- **Crop-then-upscale and upscale-then-crop now work in any order (#114, #124)**
  Aligned with [ADR 0004 Addendum 1 "Stages aren't strictly ordered"](docs/adr/0004-preprocess-manifest.md): the preprocess worker no longer skips just because the manifest already has an entry; whether to run the model is decided by `upscaler.SKIP_MODEL_RATIO` based on source pixels.

  - Explicitly selecting a cropped derivative and clicking upscale → goes through (previously skipped by mode='all').
  - Upscaling an already-upscaled output again → goes through.
  - The "Upscale all N" button's N is now the full count of images in the current grid (including cropped derivatives), not just un-derived downloads.
- **Tag editing's batch action panel reworked into rows + sortable headers + inline confirmation (#113)**
  The batch action panel is rearranged per design V2: the four operations (add / remove / replace / dedup) each take a row, with buttons aligned to one vertical line.

  - Add / remove each have their own input, no longer sharing one input box that fights over ownership.
  - The "prefix / suffix" toggle lives only inside the "add" row, so its scope is clear.
  - Only "add" is a filled primary; remove / replace / dedup are all outline; remove uses error coloring (red border + red text, no longer filled).
  - The tag distribution panel's sort dropdown becomes clickable column headers — clicking the same column toggles direction, switching columns defaults to desc, and the active column shows a ↑ / ↓ arrow.
  - The inline `×` / `✎` to delete / edit a single tag is aligned with the batch operations, all going through the confirm modal that shows the exact number of images affected.
  - Entering single-image editing auto-collapses the BulkActionBar so the sidebar makes room for the TagEditor.
- **Inference hot-swaps same-structure LoRAs + canceling a generation keeps the daemon + generation parameters persist (#114)**
  Three behavior improvements for the Generate page's inference daemon:

  - **Hot-swap same-structure LoRAs**: switching to a LoRA with the same base model / same structure no longer restarts the daemon — it switches instantly.
  - **Canceling a generation keeps the daemon**: after a single generation is canceled the daemon stays alive and reusable, no longer killed by mistake.
  - **Generation parameters persist**: after a page refresh the prompt / sampling parameters / selected LoRA and more no longer reset.
- **The outputs page merges "Download" and "Export" into a single button + a dialog with two choices (#128)**
  The outputs page drops the standalone "Export" button; clicking "Download" opens `OutputsDownloadDialog` with a radio choice (local zip / export to the data_exports directory). It matches the sidebar's `ExportBundleDialog` in style, but without the unrelated dataset / regularization set / caption / config options.
- **Training save fields get unit suffixes: `save_every_epochs` / `save_state_every_steps` (#128)**
  Two easily-confused fields are renamed:

  - `save_every` → `save_every_epochs` (the epoch interval for saving a LoRA checkpoint).
  - `save_state_every` → `save_state_every_steps` (the step interval for saving training state).

  `migrate_legacy_save_keys` (a pydantic before-validator + `bootstrap.apply_yaml_config`, two layers) migrates old yaml / old presets automatically, with no manual file edits. It also corrects the `save_state_every_epochs` description's wrong note about "whichever of X comes first" (the two state saves are actually judged independently, and their filenames carry `step{N}` / `epoch{N}` so they don't collide).
- **`grad_clip_max_norm` default 0 → 1.0, to prevent LoRA mode collapse (#125)**
  The training default for gradient clipping changes from "disabled" to `1.0`.

  A recent Qwen-image LoRA training mode-collapsed around step ~1220 (loss spiked from 0.1 to 0.9 ~ 1.0, prediction latent std shrank from 0.99 to 0.68). After a cross-direction audit by 5 algorithm experts, `grad_clip = 0` was confirmed to amplify risk across several regimes — Prodigy + ScheduleFree / DoRA + LoKr / bf16 + flow matching — and mainstream trainers (SimpleTuner / OneTrainer / ai-toolkit) all default to 1.0. The schema description was rewritten to explain the parameter's behavior and scenario-based recommended values.

### Improved

- **Dataset image grid gets virtual scrolling + a dominant-color placeholder that fades in (#113)**
  ImageGrid switches to virtual scrolling, so large datasets (hundreds to thousands of images) no longer stutter when you scroll. Before a thumbnail loads, a block of the image's dominant color holds its place and fades in once it arrives.
- **Inference daemon prewarms on startup to avoid a multi-minute stall loading text encoders (#128)**
  Users reported the inference daemon stalling for 6+ minutes at `loading text encoders`. py-spy traced the call chain `transformers.generation.candidate_generator → sklearn → scipy.special.__init__` — a scipy cold import takes minutes on Windows + Python 3.13 with GB-scale models already loaded (system RAM under pressure).

  The fix imports `transformers.generation` early when the daemon starts, so loading the text encoder is instant.
- **Tag editing intercepts navigation with a confirmation when there are unsaved changes (#128)**
  Previously only `beforeunload` caught closing the browser / refreshing, and in-app React-Router navigation silently lost changes. The rework:

  - App.tsx migrates from classic `BrowserRouter` to `createBrowserRouter + RouterProvider` (DataRouter), unlocking v6 `useBlocker`.
  - TagEdit uses `useBlocker(dirty)` + useDialog with a red confirm (discard / stay) to intercept.
  - The Sidebar + Topbar are pulled into a RootLayout wrapping the business routes in `<Outlet />`, structurally equivalent.

### Fixed

- **Fixed 4 UI scroll overflows (queue / monitor / queue detail / project card) (#110)**
  In 4 places the parent's `overflow-hidden` locked the viewport height but the child lacked `flex-1 min-h-0`, so overflowing child content was clipped by the parent and `overflow-y-auto` never triggered:

  - `/queue` couldn't scroll down once tasks overflowed the viewport.
  - `/tools/monitor` clipped panel content beyond the viewport while a task was running.
  - `/queue/:id#monitor` had the same issue (it shares MonitorDashboard).
  - `/queue/:id#overview` couldn't scroll a long error_msg / path.

  Plus the project home card grid lacked `auto-rows-fr`, so a card with a note stretched the other cards in its row out of alignment.
- **Queue hold / release / reorder endpoints occasionally returned a 422 error (#114)**
  A FastAPI route ordering problem: static paths like `/api/queue/hold` / `/api/queue/release` / `/api/queue/reorder` were captured by the `/api/queue/{task_id}` parameter route, returning 422 unprocessable entity. Fixed by reordering the routes.
- **Internal protocol lines no longer leak into the training log display + auto-backup logs are categorized separately (#114)**
  - Lines starting with `__EVENT__:` (the inter-process protocol) are no longer shown as training logs.
  - The log messages for automatic epoch backups and manual step / epoch saves are separated, no longer sharing one line (it used to be unclear which kind of save triggered).
- **Regularization pre-generation hidden from the queue + logs moved to the config tab + attention backend auto-selected (#114)**
  - The regularization-set pre-generation task is hidden from the Queue page (it used to mix in with the training task list).
  - Pre-generation logs moved into the version config tab (they used to scatter into the main queue logs).
  - The attention backend no longer requires manual selection — it's auto-detected based on the installed libraries.
- **Queue page shows only the current task's training status + nested training output renders correctly (#114)**
  The Queue page's main row no longer mixes in training metrics from historical tasks; the nested output under a task (per-epoch sample images / intermediate state) renders grouped correctly to its owning task, no longer interleaved out of place.
- **Generate page dataset prompt picker didn't clear selected tags when closed (#128)**
  The Generate page's dataset prompt picker previously only closed the UI on `×` without touching the `datasetPick` state, so the selected caption.tags were still stitched into the prompt by `handleGenerate` — you thought nothing was selected, but it was still in effect. Fixed so `onClose` also calls `setDatasetPick(null)`.
